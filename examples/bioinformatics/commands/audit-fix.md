对照 plan.md 执行穷尽式审计-修复一体化循环。不要只做计划，立即执行修复。

## 边界（CONSTRAINT）

- 最多执行 5 轮循环
- 每轮按 P0→P1→P2→P3 优先级顺序修复
- P0/P1 全部清零且无新问题时提前终止
- 每轮开始前输出当前轮次和剩余问题数
- 审计后立即修复，不只做计划
- 每轮修复后建议提交（如果项目使用 git 且有未提交的修复）

  ### 在此边界内追求（ASPIRATION）

  以下所有追求不得违反上方边界。

  - 穷尽式覆盖：6 个审计维度全部检查，不遗漏
  - 追溯源头：数据不一致时追溯到原始文件修正，不做表面修补
  - 高效迭代：每轮尽可能修复多个问题，减少总轮次
  - 修复不引入新问题：每轮重新审计验证

    #### 可自主决定（FREEDOM）

    以下选择空间在上方边界和追求方向内自主发挥。

    - 具体修复方案（重跑脚本 / 修改参数 / 重新生成图表）
    - 审计维度的检查顺序
    - P2/P3 问题是否在本轮修复或推迟到后续轮次

## 问题分级

- **P0 — 数据完整性**: 缺失结果、数据丢失、关键文件损坏
- **P1 — 方法正确性**: 参数错误、工具版本不对、基因组版本不匹配
- **P2 — 完整性缺失**: 部分分析未完成但不影响核心结论
- **P3 — 润色优化**: 图表美化、报告格式、命名规范

## 审计维度

1. **完整性**: plan.md 中每个分析步骤是否都有对应结果文件
2. **数据准确性**: 结果文件中的数据是否与原始数据/中间文件一致（抽样验证）
3. **分析准确性**: 统计方法、参数设置、阈值选择是否正确
4. **方法合理性**: 工具/流程/参考基因组版本是否合适
5. **图表-数据一致性**: 图表中展示的数据是否与结果文件吻合
6. **基因组版本**: 参考基因组/注释版本是否与 plan.md 要求一致

## 输出格式

```
| 轮次 | P0 | P1 | P2 | P3 | 状态 |
|------|----|----|----|----|------|
| R1   | 3  | 5  | 2  | 4  | 修复中 |
| R2   | 0  | 2  | 2  | 4  | 修复中 |
| R3   | 0  | 0  | 1  | 3  | 通过 ✅ |
```

剩余未修复问题（如有）及原因说明。
